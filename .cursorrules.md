## 2. 项目架构总览

### 2.1 技术栈

| 技术 | 选择 | 理由 |
|------|------|------|
| **游戏引擎** | Cocos Creator 3.8+ | 微信小游戏官方推荐，TypeScript 原生支持 |
| **开发语言** | TypeScript | 类型安全，AI 代码生成友好 |
| **UI 框架** | Cocos UI 系统 + 自定义组件 | 轻量，无需第三方库 |
| **动画** | Cocos Tween + 粒子系统 | 适合矢量风格的程序化动画 |
| **状态管理** | 自定义 EventBus + 单例 StateManager | 游戏状态集中，响应式更新 |
| **配置表** | JSON 静态配置 | 通过 `ConfigTable` 加载，支持热更新 |
| **测试** | Node.js + ts-node | `MemoryConfigLoader` + `MemoryStorageAPI` 本地运行 |
| **后端** | 微信云开发 | 云函数 + 云数据库，免服务器运维 |

### 2.2 目录结构（强制）

```

  systems/       BattleSystem, EconomySystem, ShopSystem, InventorySystem,

  heroes.json    角色配置表
  enemies.json   敌人配置表
  mods.json      改装词条配置表
  events.json    事件节点配置表
```

### 2.3 核心设计原则（不可违背）

1. **事件驱动**：所有系统间通信通过 `EventBus`，禁止直接引用其他 System 的内部状态。
2. **依赖注入**：每个 System 的依赖通过构造函数参数传入，禁止全局变量（`window`/`global`）。
3. **配置驱动**：所有数值（CD、伤害、价格、敌人属性）必须来自 JSON 配置表，禁止代码硬编码。
4. **接口优先**：每个 System / Model 必须有 `I` 前缀的接口定义，便于 Mock 测试。
5. **文件行数限制**：每个文件 ≤ 500 行，核心循环文件可放宽至 800 行。每个函数 ≤ 80 行。
6. **测试必过**：每新增一个 System 或修改核心逻辑，必须同步修改/新增 `__test__/*.test.ts` 测试，且 `npm test` 全部通过。
7. **懒更新**：CDSystem 使用 `Set<ItemInstance>` 只追踪有 CD 的装备，避免每帧遍历全部 6 格。
8. **对象池**：飘字、粒子、特效在 update 循环中禁止创建新对象，必须复用。
9. **逻辑与表现分离**：`ui/*.ts` 等不得 `import 'cc'`；Cocos 绑定只在 `*View.ts` 组件。
10. **微信 API 抽象**：所有微信 API 调用必须经 `StorageAPI` / `WeChatBridge` 封装，测试环境使用 `MemoryStorageAPI`。

## 3. MVC 控制规范（方案 C — 务实版）

### 3.1 四层架构定义

本项目采用 **View → Controller → Model → Data** 严格分层。当前阶段为务实混合模式：

```
┌─────────────────────────────────────────────┐
│  View 表现层（Cocos 组件）                    │
│  *View.ts ── 绑定 Node、监听点击、播放动画     │
│  通过 GameAppHolder 获取 GameApp 并调用公开方法 │
│  通过 EventBus 监听状态变更更新 UI             │
└─────────────────────────────────────────────┘
                      ↓ 调用公开方法 / 用户输入
┌─────────────────────────────────────────────┐
│  Controller 控制层（systems/）                 │
│  接收输入、执行业务逻辑、修改 Model 状态         │
│  通过 EventBus 发布状态变更事件               │
│  禁止直接操作 UI 节点、禁止 import 'cc'       │
└─────────────────────────────────────────────┘
                      ↓ 修改 / 读取
┌─────────────────────────────────────────────┐
│  Model 数据层（models/ + 部分内聚状态）         │
│  纯数据结构 + 有限 setter 逻辑                │
│  禁止包含复杂业务规则（如伤害公式、经济计算）    │
└─────────────────────────────────────────────┘
                      ↓ 读取 / 写入
┌─────────────────────────────────────────────┐
│  Data 持久层（config/ + SaveManager）           │
│  JSON 配置、存档、云同步、网络请求            │
│  禁止包含业务规则                            │
└─────────────────────────────────────────────┘
```

### 3.2 各层职责与禁止事项

| 层级 | 文件位置 | 职责 | 允许做的事 | 绝对禁止 |
|:---|:---|:---|:---|:---|
| **View** | `*View.ts`（Cocos 组件） | 渲染、动画、输入监听、播放音效 | 通过 `GameAppHolder` 获取 `GameApp` 并调用**公开方法**；监听 `EventBus` 事件更新 UI | 直接修改 `GameState.instance.xxx = yyy`；直接调用 `BattleSystem._triggerItem()` 等私有方法；直接 `import` 任何 `systems/` 或 `models/` 模块 |
| **Controller** | `systems/*.ts` | 业务逻辑、数据校验、流程编排、数值计算 | 调用 `models/` 的 getter/setter；读取 `ConfigTable`；通过 `EventBus` 发布事件 | 直接 `import 'cc'`；直接操作 Cocos 节点；直接调用 `wx.*` API；直接修改其他 System 的私有字段 |
| **Model** | `models/*.ts` | 纯数据定义、有限状态变更 | 定义数据结构；简单 setter 中触发 `EventBus.emit` | 包含复杂业务规则（如伤害公式、经济计算） |
| **Data** | `config/*.ts` `SaveManager` `StorageAPI` | 配置加载、存档读写、网络请求 | `JSON.parse`、localStorage、wx.cloud | 包含业务规则；硬编码数值（除安全线常量） |

### 3.3 跨层通信规则

1. **View → Controller**：通过 `GameAppHolder.getGameApp()` 获取 `GameApp` 实例，调用其公开方法。禁止直接 `import` 任何 `systems/` 或 `models/` 文件。
2. **Controller → Model**：通过 `models/` 的接口方法读取/修改数据。禁止直接修改 `models/` 内部私有字段。
3. **Controller → Controller**：通过 `EventBus` 或公共接口方法通信。禁止直接持有对方实例并读取私有字段。
4. **Controller → View**：通过 `EventBus` 发布事件，View 层监听事件并更新显示。禁止 Controller 直接操作 Cocos 节点。
5. **跨场景状态保持**：`GameAppHolder.set/getGameApp()`，场景切换用 `director.loadScene(SceneNames.xxx)`。

### 3.4 示例：装备购买流程的 MVC 调用链

```typescript
// === View：ShopSceneView.ts（Cocos 组件）===
// 玩家点击购买按钮
onBuyBtnClick(itemId: string): void {
  const app = GameAppHolder.getGameApp();   // 获取 Controller
  app.buyItem(itemId);                      // 调用 Controller 公开方法
}

// === Controller：GameApp.ts / InventorySystem.ts ===
buyItem(itemId: string): PurchaseResult {
  const price = this._configTable.getItem(itemId).price;
  if (this._economy.gold < price) {
    return { success: false, reason: 'gold_insufficient' };
  }
  this._economy.spendGold(price);            // 修改 Model 状态
  this._inventory.addItem(itemId);         // 修改 Model 状态
  this._eventBus.emit('gold_changed', { gold: this._economy.gold });
  return { success: true };
}

// === View：ShopSceneView.ts（再次）===
// 监听 gold_changed 事件，更新金币显示
onGoldChanged(payload: { gold: number }): void {
  this.goldLabel.string = `${payload.gold}`;
}
```

### 3.5 MVC 自检清单（AI 生成代码后必须核对）

- [ ] **View 层**：是否只通过 `GameAppHolder` / `EventBus` 交互？有没有偷偷 `import '../systems/BattleSystem'`？
- [ ] **Controller 层**：是否只通过 `models/` 接口修改数据？有没有直接 `import 'cc'` 或操作 Cocos 节点？
- [ ] **Model 层**：是否只包含数据结构和简单 setter？有没有把伤害公式、经济计算塞进 Model？
- [ ] **Data 层**：所有数值是否来自配置表或存档？有没有硬编码业务规则？
- [ ] **跨层引用**：是否存在 `SystemA` 直接读取 `SystemB._privateField` 的情况？

### 3.6 架构演进决策：本项目不做全局 MVC 重构

**当前架构已足够支撑完整上线（Phase 1~3）。** 原因：
- 本项目为**中小体量微信小游戏**（单局 5-8 分钟，纯 PvE，5 角色，60+ 装备）
- 场景切换简单（Battle ↔ Shop），当前 `GameAppHolder + EventBus` 模式已覆盖全部需求
- 现有 `systems/` 混合逻辑+数据模式，文件规模可控（500-800 行），完全可维护

**明确标记：以下情况** **才** **考虑 MVC 重构或深度拆分**：

| 触发条件 | 说明 | 阶段 |
|:---|:---|:---|
| 新增**异步 PvP** 或**实时联机**模式 | 需要严格的客户端-服务端状态同步，混合模式难以保证一致性 | 非当前项目范围 |
| 角色数量超过 **20+** 或装备数量超过 **200+** | 单个 `BattleSystem` 超过 1000 行且无法通过内部拆分解决 | 当前项目上限 60+ 装备，不会触发 |
| 需要支持 **MOD 社区** 或 **玩家自定义装备** | 需要运行时动态加载和隔离第三方脚本 | 非当前项目范围 |
| 团队规模扩大到 **5+ 人** 同时维护同一 System | 需要更严格的边界隔离以减少冲突 | 当前团队 3-4 人，不会触发 |

**在此之前，只需做到：**
1. 新增代码严格遵循 MVC 分层，不反向污染（View 不直接改 Model，Controller 不直接操作 UI）
2. 单个 System 超过 800 行时，优先做**内部职责拆分**（提取辅助类、工具函数），而非全局重构
3. 保持现有 `systems/` + `ui/` 目录结构，不迁移为 `controllers/` + `views/`

## 4. 增量开发协议（The Incremental Protocol）

### 4.1 黄金规则：一次只做一个功能点

AI 禁止在单次回复中实现超过 **一个端到端功能**（如：不能同时写「升星系统」和「改装系统」）。

### 4.2 五步循环（每个功能点）

#### Step 1: 接口契约（Contract）
**AI 任务**: 只定义数据结构和 `I` 前缀接口，不写逻辑。

```typescript
// 示例：升星系统接口
export interface IStarUpgradeConfig {
  maxStar: number;           // 最大 3 星
  statMultiplier: number[];    // [1, 1.5, 2] 对应 1/2/3 星
  modSlotUnlock: number[];   // [0, 1, 2] 对应解锁改装槽数
}

export interface IItemInstance {
  instanceId: number;
  configId: string;
  star: number;              // 1~3
  modSlots: IModSlot[];
}
```

**人类动作**: 确认字段是否满足需求，调整边界值。

#### Step 2: 测试先行（Test First）
**AI 任务**: 基于契约写测试，此时测试应该**失败**（因为还没实现）。

```typescript
// 示例：升星测试
function testStarUpgradeSuccess(): void {
  resetTestEnv();
  const app = createTestApp();
  const item = app.inventory.addItem('pencil_sharpener');
  const result = app.inventory.upgradeStar(item.instanceId);
  assert(result.success === true);
  assert(item.star === 2);
  assert(item.modSlots.length === 1);
  console.log('  ✓ 升星成功，解锁 1 个改装槽');
}

function testStarUpgradeMaxStar(): void {
  resetTestEnv();
  const app = createTestApp();
  const item = app.inventory.addItem('pencil_sharpener');
  app.inventory.upgradeStar(item.instanceId);   // 2 星
  app.inventory.upgradeStar(item.instanceId);   // 3 星
  const result = app.inventory.upgradeStar(item.instanceId); // 尝试 4 星
  assert(result.success === false);
  assert(result.reason === 'max_star_reached');
  console.log('  ✓ 已达最大星级，拒绝升级');
}
```

**人类动作**: 运行测试，确认确实失败（Red 状态）。

#### Step 3: 最小实现（Minimal Implementation）
**AI 任务**: 只写让测试通过的最少代码，**禁止**过度设计。

```typescript
// 示例：最小实现
class InventorySystem {
  upgradeStar(instanceId: number): StarResult {
    const item = this._findItem(instanceId);
    if (!item) return { success: false, reason: 'item_not_found' };
    if (item.star >= 3) return { success: false, reason: 'max_star_reached' };
    item.star++;
    item.modSlots = new Array(this._getModSlotCount(item.star)).fill(null);
    this._eventBus.emit('item_star_up', { instanceId, star: item.star });
    return { success: true };
  }
}
```

**人类动作**: 运行测试，确认通过（Green 状态）。

#### Step 4: 边界加固（Harden）
**AI 任务**: 补充边界处理、错误提示、日志，但不改变核心逻辑。

- 检查 `instanceId` 是否合法
- 检查重复装备是否在同一场合（背包/装备栏）
- 补充 `Logger.info()` 记录升星操作
- 添加 TODO 标记后续需要接入的改装系统

**人类动作**: 运行测试，确认全绿；手动试几个异常输入。

#### Step 5: 提交锁定（Commit Lock）
**AI 任务**: 生成 commit message，人类执行提交。

```
feat(inventory): add star upgrade with mod slot unlock

- Implement InventorySystem.upgradeStar() up to 3 stars
- Add stat multiplier and mod slot unlock per star level
- Cover with 2 unit tests (happy path, max star boundary)
```

**人类动作**: `git add . && git commit -m "..."`，打 tag 可选。

## 5. 代码质量标准（Quality Gates）

### 5.1 文件与函数约束（铁律）

```
1. 每个文件 ≤ 500 行（核心循环文件可放宽至 800 行）
2. 每个函数 ≤ 80 行（辅助函数 ≤ 30 行）
3. 所有外部依赖通过参数注入（禁止全局变量 window/global）
4. 所有异步操作必须有 try-catch + 日志记录
5. 所有用户操作必须有防抖/节流（300ms）
6. 每个 System 必须配套单元测试（.test.ts）
7. 所有数值使用配置表，禁止硬编码（magic number 零容忍）
8. 所有类必须有接口定义（I 前缀），便于 Mock 测试
9. 复杂逻辑必须写注释说明「为什么」
10. 禁止在 update 循环中创建新对象（必须对象池）
11. 禁止在 System 之间直接引用内部状态（必须通过 EventBus 或接口方法）
12. 所有配置表字段变化必须同步更新测试用的 MemoryConfigLoader 数据
```

### 5.2 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 接口 | `IXxx` | `IBattleSystem`, `IItemConfig`, `IHeroState` |
| 类 | `Xxx` | `BattleSystem`, `ItemConfig`, `SaveManager` |
| 事件名 | 小写下划线 | `item_triggered`, `gold_changed`, `battle_end` |
| 配置表字段 | camelCase | `baseCD`, `heroAffinity`, `maxStar`, `effectType` |
| 文件路径 | camelCase | 与主类名一致，如 `BattleSystem.ts` |
| 常量 | UPPER_SNAKE_CASE | `CD_MIN`, `MAX_FRAME_DAMAGE`, `SLOT_COUNT` |
| 私有方法 | 下划线前缀 | `_triggerItem()`, `_applyHaste()` |
| 私有属性 | 下划线前缀 | `_eventBus`, `_items`, `_currentGold` |
| 类型别名 | `T` 前缀 | `TItemRarity`, `TBattleResult` |

### 5.3 事件命名规范（强制）

所有事件名必须遵循以下命名约定，并在「附录 A：事件清单」中登记：

| 场景 | 命名模式 | 示例 |
|------|---------|------|
| 状态变更 | `{field}_changed` | `hp_changed`, `gold_changed`, `item_star_up` |
| 用户请求 | `{action}_requested` | `buy_item_requested`, `attack_requested`, `rescue_charge_requested` |
| 流程结束 | `{phase}_ended` | `battle_ended`, `round_ended`, `shop_closed` |
| 加速/触发 | `haste_applied`, `item_triggered` | `haste_applied`, `item_triggered`, `damage_dealt` |

- 事件 payload 必须定义接口（如 `IItemTriggeredPayload`），禁止用 `any`。
- 触发事件时，必须确认数据已准备好（避免发送半成品状态）。
- 监听事件时，必须检查 payload 是否为 `undefined`（EventBus 允许无参触发）。

### 5.5 安全红线（不可妥协）

AI 绝对禁止以下代码，发现立即重写：

| # | 禁止行为 | 正确做法 |
|:---:|:---|:---|
| 1 | 代码中出现密码、API Key、Token | 读取环境变量或 `wx.cloud.callFunction` |
| 2 | SQL 字符串拼接（如有后端） | 使用参数化查询 / ORM |
| 3 | 直接返回数据库异常给客户端 | 包装为统一错误格式，隐藏内部细节 |
| 4 | 用户上传文件不校验后缀/大小 | 白名单后缀 + 大小限制 + 重命名存储 |
| 5 | 在日志中打印用户敏感信息 | 脱敏处理，只打印 ID 或哈希 |
| 6 | `eval()` / `exec()` 处理用户输入 | 使用 JSON / 专用解析器 |
| 7 | 前端直接信任传来的 userId 做权限判断 | 服务端从 Session/JWT 重新解析 |
| 8 | 微信 API 直接硬编码在逻辑层 | 必须经 `WeChatBridge` / `StorageAPI` 抽象 |
| 9 | 在 `update` 循环中创建新对象 | 使用对象池（飘字、粒子、特效） |
| 10 | 数值硬编码在逻辑代码中 | 全部提取到 `resources/config/*.json` |

## 6. Cocos Creator 特定规则

### 6.1 生命周期纪律

- **禁止在 `constructor` 中写逻辑**，全部移到 `onLoad()`。
- `onDestroy()` 中必须清理：
  - `EventBus.off()` 取消所有本组件注册的事件
  - `this.unschedule()` 取消所有定时器
  - 释放动态加载的资源（`assetManager.releaseAsset()`）
- **禁止 `setInterval` / `setTimeout`**，统一用 `this.schedule(callback, interval)`。
- 在 `onDestroy` 中 `this.unschedule(callback)`，场景切换前清理所有未触发的 schedule。

### 6.2 节点查找与引用

| 优先级 | 方式 | 说明 |
|--------|------|------|
| **首选** | `@property` 编辑器绑定 | 属性检查器拖拽，最稳定 |
| **次选** | `find('Canvas/GameLogic')` | 路径查找，仅在场景结构固定时使用 |
| **避免** | `getChildByName()` | 易因节点改名失效 |
| **禁止** | 硬编码节点路径字符串分散在多个文件中 | 集中管理路径常量 |

### 6.3 资源加载

- **使用 `assetManager.loadBundle()` 和 `assetManager.load()`**。
- **禁止已废弃的 `resources.load()`**（Cocos 3.8 已废弃）。
- 动态加载的资源必须在 `onDestroy` 中释放（`assetManager.releaseAsset()`）。
- 场景切换时调用 `assetManager.releaseUnusedAssets()`。

### 6.4 Cocos 编辑器教程输出格式（强制）

向用户说明 **Cocos 编辑器挂接、预制体、场景配置** 时，必须遵守 `.cursor/rules/cocos-editor-tutorial-format.mdc`，包括但不限于：

1. **标明面板**：层级管理器、资源管理器、属性检查器等，以及打开的 `.scene` / `.prefab` 路径。
2. **创建项四要素**：父节点、**命名**、创建**类型**（菜单路径）、需添加的**组件**。
3. **赋值**：写明在 **属性检查器** 中哪一组件的哪一属性配置何值。
4. **拖放方向**：从哪拖到哪（层级 → 属性检查器槽位）。
5. **拖入类型（必写）**：每条拖放须标明 **Label 组件**、**Sprite 组件**、**节点**、**{脚本名} 组件** 或 **资源**；多条绑定用三列表格（属性名 | 层级路径 | 拖入类型）。

禁止模糊表述（如「拖进去」「绑一下」「拖到 Icon 上」）而不指明面板、起止对象与 **组件类型**。

### 6.5 动画与特效

- 简单动效（飘字、缩放、变色）用 `Tween` 代码实现。
- 装备触发特效用 `Animation` 组件 + SpriteSheet（4-8 帧）。
- 复杂粒子用 `ParticleSystem2D`，必须开启 **烘培模式**（Baked）。
- **同屏特效数量 ≤ 5 个，单特效粒子数 ≤ 50**。
- 纹理尺寸必须是 2 的幂次方（如 128×128、256×256）。
- UI 图集使用 TexturePacker 打包，减少 DrawCall。

## 7. 内存安全与性能红线

### 7.1 对象池（强制）

以下对象必须使用对象池复用，禁止运行时 `new` 或 `instantiate`：

- 伤害数字飘字
- 特效节点
- 商店商品卡片
- 战斗日志条目
- 子弹/投射物（如有）

### 7.2 事件监听安全

- `EventBus.on()` 必须在 `onDestroy()` 中配对 `EventBus.off()`。
- 使用 `this.node.on()` 时，同样在 `onDestroy` 中 `this.node.off()`。
- **禁止匿名函数监听后无法移除**（必须保存回调引用）。

### 7.3 性能红线（绝对不可突破）

| 指标 | 上限 | 说明 |
|------|------|------|
| DrawCall | ≤ 100 | 通过图集合批控制 |
| 同屏节点数 | ≤ 500 | 包含隐藏节点 |
| 单特效粒子数 | ≤ 50 | 超过必须开烘培 |
| 同屏粒子总数 | ≤ 200 | 所有特效叠加 |
| 音频同时播放 | ≤ 8 个 | 超出自动丢弃 |
| 动态刚体 | ≤ 50 | 物理计算上限 |
| 静态刚体 | ≤ 200 | 碰撞检测上限 |
| 装备 CD 硬下限 | 0.3 秒 | 防止帧率崩溃 |
| 单帧装备触发上限 | 1 次/装备 | 防止死循环 |

## 8. 微信小游戏适配

### 8.1 包体控制

- 首包资源（主包）< 4MB
- 总包 < 20MB
- 分包策略：核心代码 + 首屏资源在主包，其余按需加载
- 首局加载 > 3 秒时，自动切换「极速模式」
- 未加载资源显示占位图标（问号箱子），点击后下载

### 8.2 音频规范

| 项目 | 要求 |
|------|------|
| 格式 | MP3 |
| 声道 | 单声道（Mono） |
| 采样率 | 44100Hz |
| 短音效（< 3秒） | 预加载 |
| 背景音乐 | 流式加载（Streaming） |
| 同时播放上限 | 8 个（超出自动丢弃） |
| 背景音乐音量 | 30%（不干扰音效） |
| iOS 静音模式 | 检测 AudioContext 状态，用户交互后恢复播放 |

### 8.3 存储规范

- 使用 `wx.setStorageSync` / `wx.getStorageSync`
- 做好降级：检测不到微信环境时用 `localStorage`
- **单键值存储 < 1MB**
- 存档数据必须 JSON 序列化
- 所有微信 API 调用必须经 `StorageAPI` / `WeChatBridge` 抽象层，测试环境使用 `MemoryStorageAPI`

### 8.4 网络与加载

- 首局加载 > 3 秒时，自动切换「极速模式」（只加载主包 + 10 件核心装备 + 5 个角色）
- 未加载的装备显示为「问号箱子」+ 文字名称，点击后提示「下载中」
- 分包策略：
  - 主包（<4MB）：核心框架、主界面、前 3 回合资源
  - 分包1（<8MB）：战斗资源、角色动画、特效、音效
  - 分包2（<8MB）：后期装备、BOSS 资源、背景音乐

## 10. 版本控制纪律（Git Hygiene）

### 10.1 提交节奏

- **每完成一个 Step（契约/测试/实现/加固）就提交一次**
- **不要等整个功能做完才提交**
- **提交前运行测试，确保全绿**

### 10.2 Commit Message 格式

```
<type>(<scope>): <一句话描述>

<可选：为什么做这个变更，解决了什么问题>
```

**Type 枚举**:
- `feat`: 新功能
- `fix`: Bug 修复
- `test`: 补充测试
- `refactor`: 重构（功能不变）
- `docs`: 文档/注释
- `chore`: 配置/依赖/工具链

**Scope 示例**: `battle`, `shop`, `hero`, `inventory`, `config`, `ui`, `test`

**好例子**:
```
feat(inventory): add star upgrade with mod slot unlock

- Implement up to 3-star upgrade
- Add stat multiplier per star level
- Cover with 2 unit tests
```

### 10.3 回滚安全

- 每次大功能开始前，基于上一个稳定 commit 创建分支：`git checkout -b feature/xxx`
- 如果 AI 把代码搞乱了，人类随时可 `git reset --hard HEAD` 回到上一个稳定点
- 重要里程碑打 tag：`git tag -a v0.3 -m "star upgrade complete"`

## 13. 常见陷阱与应对（AI 自检清单）

### 13.1 项目特有陷阱（大巴扎必查）

| 陷阱 | 说明 | 规避方法 |
|------|------|----------|
| **加速死循环** | 装备 A 加速 B，B 加速 A，导致每帧无限触发 | `CD_MIN` 硬下限 + `triggeredThisFrame` 单帧保护 + deferredQueue 跨帧队列 |
| **快照引用泄漏** | 快照存储了 `ItemInstance` 的引用而非值，恢复后修改原数据 | 快照必须深拷贝（`JSON.parse(JSON.stringify(...))`） |
| **instanceId 冲突** | 从快照恢复后新创建的装备 ID 与快照中的重复 | 恢复时调用 `syncInstanceIdCounterFromSave()` 同步计数器 |
| **配置表不同步** | `resources/config/*.json` 更新了，但测试用 `MemoryConfigLoader` 数据没同步 | 修改配置后必须同步更新测试数据，测试失败时首先检查此问题 |
| **EventBus 内存泄漏** | 监听事件后未 off，导致闭包持有引用无法 GC | 场景销毁时调用 `eventBus.clear()` 或精确 off |
| **wx API 在测试中报错** | 测试环境没有 `wx` 对象 | 所有微信 API 调用必须通过 `StorageAPI` 等抽象层，测试使用 `MemoryStorageAPI` |
| **iOS 静音模式** | WebAudio 在 iOS 静音模式下不播放 | 检测 AudioContext 状态，用户首次交互后恢复播放 |
| **分包加载失败** | 资源路径错误或分包未下载 | 使用占位图机制，未加载资源显示为问号箱子 |
| **Retry 无反应** | 未 `takeRoundSnapshot` | `startRound` 开头补快照 |
| **商店金币显示默认** | `GameApp` 未就绪 | `onEnable` 从 `GameAppHolder` 取 app |
| **EPERM / 绑定失败** | `@property` 复杂类型 | 改为 `Node` + `getComponent` |
| **MVC 越层调用** | `BattleSystem` 直接操作 Cocos 节点 | 严格分层：Controller 层只通过 EventBus 通信，View 层只监听事件 |

## 15. 架构演进路线图（未来阶段约束）

以下约束**当前阶段不需要执行**，但必须在规范中记录，以便未来条件触发时 AI 知晓何时升级。AI 当前生成代码时**不强制**执行以下内容，但**禁止**引入与这些规则冲突的新架构。

### 15.1 触发条件与升级时机

| 约束项 | 当前状态 | 升级时机 | 说明 |
|--------|---------|---------|------|
| **TypeScript `strict: true`** | 【当前推荐】逐步开启 | Phase 3（性能优化与上线准备）全面开启 | 现有代码若未适配，开启会导致编译失败。建议从 `noImplicitAny` 开始逐步开启 |
| **类方法 ≤ 15 个** | 【当前推荐】软约束 | 单个 System 超过 20 个方法时内部拆分 | 不强制全局执行，超过时优先做内部职责拆分（提取辅助类） |
| **目录重命名 `controllers/` / `views/`** | 【当前不推荐】维持 `systems/` + `ui/` | 新项目或完全重写时考虑 | 迁移成本：全项目 import 路径修改 + 测试路径修改，收益几乎为零 |
| **纯 EventBus 通信（View 不持有 GameApp）** | 【当前不推荐】保留 `GameAppHolder` 模式 | 新增**异步 PvP** 或**实时联机**模式时才考虑 | 当前纯 PvE 单局场景，`GameAppHolder` 模式已足够，纯 EventBus 增加调试复杂度 |
| **Model / Controller 深度拆分** | 【当前不推荐】维持 `systems/` 混合模式 | 项目规模超过 **10 万行** 或团队 **5+ 人** 并行维护同一 System | 当前 500-800 行 / 文件，完全可控，拆分反而引入"贫血 Model"和跳转成本 |
| **确定性随机（种子化）** | 【当前推荐】框架预留 | Phase 2（社交系统与排行榜）实现 | 用于战斗回放和反作弊。当前可用 `Math.random()`，但需标记 `TODO: 替换为 seeded RNG` |
| **事件命名 `_requested` 全覆盖** | 【当前推荐】新事件优先使用 | 重构现有事件时逐步替换 | 现有 `item_purchased` 等事件无需强制改名，新事件优先使用 `buy_item_requested` |

### 15.2 演进原则

1. **不为了规范而规范**：架构升级必须有明确的业务需求驱动（如联机对战、MOD 系统），而非为了"更优雅"。
2. **新增代码优先合规**：即使旧代码未重构，新文件必须严格遵循当前规范（MVC 分层、事件命名、对象池等）。
3. **逐步替换，禁止大爆炸重构**：任何架构升级必须满足：
   - 有独立的 feature 分支
   - 测试全部通过
   - 手动验证通过（Cocos 预览 + 微信开发者工具）
   - 人类确认后才合并到 main

### 15.3 决策树（AI 自问）

当 AI 建议引入新的架构约束时，必须按以下顺序判断：

```
1. 当前业务是否需要这个约束？（如：没有 PvP，不需要纯 EventBus）
   → 否：不引入，标记为未来阶段
   → 是：继续
2. 现有代码是否已支持这个约束？（如：strict 模式是否会导致编译失败）
   → 否：标记为逐步开启，不强制立即执行
   → 是：继续
3. 引入这个约束是否会让现有测试失败？
   → 是：先修复测试，或标记为未来阶段
   → 否：可以引入
```
